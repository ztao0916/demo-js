# 产品样式规范

> 适用于 Trade-Web 后台产品原型、UI 走查、前端样式对齐。
>
> 最后更新: 2026-04-30

---

## 1. 适用范围

本规范用于统一 Trade-Web 后台页面的产品原型标注和视觉走查口径，重点覆盖项目中高频出现的页面结构：

- 查询区: `.search_card_new`
- 工具栏: `.tools_btn`
- 列表区: `.card_position.list_card` / `.list_card`
- 弹窗: `.el-dialog`
- 下拉菜单: `.el-dropdown-menu__item`
- 组合输入控件: `.form_left` / `.form_middle` / `.form_right`

本规范不替代单页面业务布局。页面特殊宽度、业务弹窗高度、特殊表格列样式等，需要保留在具体页面或组件中。

---

## 2. 原型设计基准

Trade-Web 产品原型以 Element Plus 后台组件体系为基础，并结合项目现有业务页面规范绘制。

产品侧无需关注 Vue、Vite、Pinia 等工程框架。原型设计时应关注“组件形态”和“业务场景尺寸”，优先参考以下口径：

| 场景 | 原型基准 |
| --- | --- |
| 普通后台控件 | 参考 Element Plus 的 Button、Input、Select、DatePicker、Dialog、Dropdown、Tabs、Table、Pagination、Form |
| 复杂表格能力 | 参考 Trade-Web 现有列表页和复杂表格形态，不需要按 VXE-Table / AG-Grid 内部实现细节标注 |
| 查询区、工具栏、列表区、弹窗 | 按本规范的业务场景尺寸绘制 |
| 页面特殊业务结构 | 优先参考 Trade-Web 现有页面模式 |

对产品的统一表述建议：

```text
产品原型以 Element Plus 后台组件为基础，不需要重新设计一套组件。
Trade-Web 在 Element Plus 基础上有自己的后台页面密度规范和业务场景规范，原型需优先遵守本产品样式规范。
```

## 3. 基础 Token

| Token | 规范值 | 适用说明 |
| --- | --- | --- |
| 全局圆角 | `2px` | 常规组件、按钮、输入框、VXE 常用输入/按钮 |
| 表单控件高度 | `32px` | 查询区、弹窗表单输入/下拉/日期 |
| 小按钮高度 | `24px` | 表格行内按钮、弹窗按钮 |
| Tab 高度 | `40px` | 列表区页签 header/nav/item |
| 基础字号 | `13px` | 表单输入、日期范围输入、下拉菜单项 |
| 小字号 | `12px` | 查询区按钮、工具栏按钮、表格体按钮、弹窗按钮 |
| 选中值颜色 | `#000` | `.form_left` 选中值 |

## 4. 圆角规范

| 对象 | 规范 |
| --- | --- |
| 全局组件圆角 | `2px` |
| 按钮圆角 | `2px` |
| VXE 常用输入/按钮圆角 | `2px` |
| 查询区输入/下拉/日期圆角 | `2px` |
| 工具栏按钮圆角 | `2px` |
| 表格体按钮圆角 | `2px` |
| 弹窗按钮圆角 | `2px` |

### 组合控件例外

`.form_left` / `.form_middle` / `.form_right` 属于组合输入控件规则。为了让多个控件拼接成一个整体，拼接边允许局部圆角为 `0`：

- `.form_left`: 右侧圆角可为 `0`
- `.form_middle`: 左右两侧圆角可为 `0`
- `.form_right`: 左侧圆角可为 `0`

原型标注时应标注为“组合控件整体圆角 2px，内部拼接边无圆角”，不要把内部拼接边按全局圆角处理。

## 5. 查询区 `.search_card_new`

查询区用于页面顶部筛选条件，常见结构为 `.search_card_new .search_form_new`。

| 对象 | 规范 |
| --- | --- |
| 输入框 | `min-height: 32px` |
| 下拉框 | `min-height: 32px` |
| 日期选择器 | `min-height: 32px` |
| 查询区按钮 | `min-height: 32px`，`font-size: 12px`，`border-radius: 2px` |
| Label | `line-height: 32px` |
| 日期范围输入 | `font-size: 13px` |

### 原型建议

- 查询项垂直方向按 `32px` 控件高度对齐。
- Label 与控件垂直居中，不额外增加顶部偏移。
- 查询、重置、展开/收起等按钮统一按 `32px` 高度绘制。

---

## 6. 工具栏 `.tools_btn`

工具栏用于列表上方的批量操作、导入导出、新增等页面级操作。

| 对象 | 规范 |
| --- | --- |
| 工具栏按钮高度 | `32px` |
| 工具栏按钮字号 | `12px` |
| 工具栏按钮圆角 | `2px` |

### 使用边界

工具栏按钮是页面级操作按钮，不与表格行内按钮共用高度。表格行内按钮使用 `24px`。

---

## 7. 列表区 `.card_position.list_card` / `.list_card`

列表区用于承载表格、页签、分页、批量操作提示等内容。

### 表格体按钮

| 对象 | 规范 |
| --- | --- |
| 表格行内按钮高度 | `24px` |
| 表格行内按钮字号 | `12px` |
| 表格行内按钮圆角 | `2px` |

表格行内按钮通常用于编辑、删除、查看、同步等行级操作，应保持轻量，不使用工具栏按钮的 `32px` 高度。

### Tab

| 对象 | 规范 |
| --- | --- |
| `.el-tabs__header` | `height: 40px` |
| `.el-tabs__nav-scroll` | `height: 40px` |
| `.el-tabs__item` | `height: 40px`，`line-height: 40px` |

Tab 高度需要 header、nav、item 同步，避免视觉高度和点击区域不一致。

---

## 8. 下拉菜单 `.el-dropdown-menu__item`

| 对象 | 规范 |
| --- | --- |
| 菜单项字号 | `13px` |
| 非空项 | 拉高显示，保证可点击区域清晰 |
| 空项 | 压缩高度，避免空值占高 |

### 原型建议

- 有文本或操作内容的菜单项按正常高度展示。
- 空占位项不应占据与正常菜单项相同的高度。
- 不使用空菜单项制造视觉间距，间距应通过菜单容器或分组表达。

---

## 9. 弹窗 `.el-dialog`

弹窗用于编辑、详情、确认、批量操作等场景。

### 表单控件

| 对象 | 规范 |
| --- | --- |
| 输入框 | `height: 32px` |
| 下拉框 | `height: 32px` |
| 日期选择器 | `height: 32px` |
| Label | `line-height: 32px` |
| Switch | 按 `32px` 控件高度垂直对齐 |

### 弹窗按钮

| 对象 | 规范 |
| --- | --- |
| 弹窗按钮高度 | `24px` |
| 弹窗按钮字号 | `12px` |
| 弹窗按钮圆角 | `2px` |

### 使用边界

弹窗表单控件使用 `32px`，但弹窗底部按钮使用 `24px`。不要因为同处于弹窗内就把按钮高度提升到 `32px`。

## 11. 当前实现差异

以下内容是目标规范与当前实现之间的已知差异，后续如果进行 CSS 落地，需要单独验证。

| 区域 | 当前实现 | 目标规范 | 说明 |
| --- | --- | --- | --- |
| `.list_card` Tab | `32px` | `40px` | `src/styles/element-ui.scss` 中 `.el-tabs__header`、`.el-tabs__nav-scroll`、`.el-tabs__item` 当前为 `32px` |
| 自研选择组件内部圆角 | 部分为 `4px` / `5px` | `2px` | `ZSelect`、`ZCascader` 内部样式后续落地时需单独适配 |

本次规范建设只记录差异，不修改前端样式。

## 12. 原型标注建议

产品原型应按页面区域标注，而不是按 Element Plus 内部类名标注。

推荐标注方式：

| 原型区域 | 标注口径 |
| --- | --- |
| 查询区 | 查询控件高度 `32px`，label `32px` 垂直居中，按钮 `32px / 12px / 2px` |
| 工具栏 | 按钮 `32px / 12px / 2px` |
| 列表区 | 行内按钮 `24px / 12px / 2px`，Tab `40px` |
| 弹窗 | 表单控件 `32px`，底部按钮 `24px / 12px / 2px` |
| 下拉菜单 | 字号 `13px`，空项压缩高度 |

当产品原型存在特殊场景时，应在原型中明确标注业务原因，避免被误认为公共规范。
